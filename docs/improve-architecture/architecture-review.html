Customers module — depth & production readiness

A read of the documented blueprint (CODEBASE.md) against the actual code (page.tsx, customers.service.ts, the mutation hooks, package.json). Vocabulary follows the module / interface / seam / depth model: the goal is to turn shallow modules (interface almost as complex as the code behind it) into deep ones (a lot of behaviour behind a small interface), for the sake of scale, testability, and AI-navigability.

The three questions you asked

Is the style good?
Yes — strong

The feature-slice + service/hook/mapper separation is genuinely deep and modern. This is a good 2026 foundation. Keep it.

Good for 2026?
Mostly

Stack is current (Next 16 / React 19 / TanStack v5 / Zod v4). But it's all client-rendered force-dynamic; it ignores Server Components / Server Actions that 2026 App Router leans on.

Production ready?
Not yet

Two blockers: the list loads every row and paginates in memory, and timestamps trust the browser clock. Both are fixable at one seam each.

Deepening candidates

1 · Push list querying into the service seam

Strong

Files: customers.service.ts · use-customers.ts · customers.keys.ts · customers-page.tsx

Problem. getActiveCustomers(client) is a shallow read: its interface can express nothing but "give me everything," so it pulls every non-deleted row and the page does the real work — search across five fields, type filter, and slicing into pages of 6 — in a useMemo in the browser. A CustomerFilters type already exists in customers.keys.ts but nothing flows through it. For a per-tenant SaaS, one busy station means the whole customer book crosses the wire on every visit.

Solution. Widen the interface, not the caller: getActiveCustomers(client, { search, type, page, perPage }) returning { rows, total }, using Supabase .ilike(), .eq(), and .range(). Wire the existing customerKeys.list(filters) as the query key so pages cache independently. The page keeps only its input state; the counting and slicing move behind the seam.

Before · shallow
service ──▶ ALL rows
    │
page.useMemo:
  • search 5 fields
  • filter type
  • slice(page,6)
↳ logic lives in the view
After · deep
service(filters) ──▶
  .ilike / .eq / .range
  └▶ { rows, total }
    │
page: just holds inputs
↳ one place to test paging

Benefits. Leverage: every future module inherits the same pageable read shape. Locality: pagination correctness (off-by-one, empty page, total count) becomes one testable service function instead of arithmetic scattered in a 260-line component. The interface becomes the test surface — you can assert paging without rendering React.

2 · Let the database own time and lifecycle

Strong

Files: customers.service.ts · customers.mapper.ts · DB migration

Problem. archiveCustomer, setCustomerStatus, and the mapper all stamp new Date().toISOString() — the browser clock. In a business where "when was this delivered / archived" is an audit fact, a wrong or skewed client clock writes a wrong fact, and a client could set it deliberately. The seam that should own "now" (Postgres) is being bypassed.

Solution. Move created_at/updated_at to default now() + an updated_at trigger, and set deleted_at = now() in SQL (or via a soft-delete RPC). The service stops sending timestamps; the mapper stops computing them. // ponytail: a DB default is fewer lines and the correct clock.

Before
browser clock
  └▶ toISOString()
    └▶ sent in payload
↳ trust the client for audit time
After
Postgres now()
  └▶ default / trigger
↳ one authoritative clock, RLS-side

Benefits. Locality: "when did it happen" has exactly one source of truth. Leverage: every table that adopts the convention gets trustworthy timestamps for free, and the service functions shrink.

3 · Collapse the five mutation hooks into one deep factory

Worth exploring

Files: use-create-customer.ts · use-update-customer.ts · use-archive-customer.ts · use-set-customer-status.ts

Problem. The four mutation hooks are the same eleven lines: grab client, grab queryClient, useMutation, invalidate lists(), toast.success, toast.error(error.message). Each hook is shallow — its interface (a mutation) is nearly its whole body. Multiply by five modules still to build and that's ~25 copies of the same wiring. Apply the deletion test: delete one hook and the "complexity" that reappears at the call site is a single useMutation config — the sign of a pass-through.

Solution. One shared useEntityMutation({ mutationFn, invalidate, successMessage }) in src/hooks that folds in the client, invalidation, and toast policy. The per-feature hooks become one-liners that supply only what differs. // ponytail: this is the rare case where an abstraction subtracts code — two real adapters (create, update) already exist, so the seam is real, not hypothetical.

Before · 4 shallow copies
create ─┐
update ─┤ each: client +
archive ─┤ qc + invalidate +
status ─┘ 2 toasts (×N modules)
After · 1 deep hook
useEntityMutation(cfg)
  └▶ create/update/…
    = 1 line each
↳ toast + cache policy tested once

Benefits. Locality: the "how do mutations behave" decision (error copy, what to invalidate, whether to toast) lives in one module. Leverage: a new module writes create/update/archive as three lines. Caution: keep the factory thin — the moment a hook needs bespoke onSuccess (e.g. detail(id) invalidation on update), let it pass overrides rather than growing flags.

4 · Close the UI drift: inline styles & the hand-rolled table

Worth exploring

Files: customers-page.tsx · customers-table.tsx

Problem. CLAUDE.md mandates Tailwind, shadcn, and TanStack Table "for every module's data table." The actual page is ~260 lines of inline style={{…}} objects and a hand-built grid; @tanstack/react-table is installed but unused here. This isn't a runtime bug — it's navigability drift: the documented pattern and the real code disagree, which is exactly the thing that misleads the next human and the next AI agent. Styling logic is also inseparable from layout, so nothing is reusable across modules.

Solution. Two honest options — pick one and make the doc match reality:
(a) Adopt the documented stack: move the table to TanStack Table + shadcn, replace inline styles with Tailwind classes / the existing CSS-var tokens.
(b) If the inline-token look is deliberate, amend CODEBASE.md/CLAUDE.md to describe what's actually done and drop the unused table dep. // ponytail: don't rewrite pretty working UI for dogma — but the doc must not lie about it.

Benefits. Locality: a shared DataTable adapter means sorting/paging/empty-state render the same in every module. Navigability: doc and code stop contradicting each other, so agents scaffolding module #2 copy the real pattern.

5 · Give Owner/Staff roles an authorization seam

Worth exploring

Files: customers.guards.ts · customer-row-actions.tsx · use-customer-owner.ts

Problem. The domain has two roles with different powers, but the module's only guard is canEditCustomer (a deleted_at check). Nothing asks "may this role archive a customer?" If RLS is the only enforcement, the UI happily shows actions a Staff user will only fail at write time — a poor experience and an easy place for the rule to be forgotten as modules multiply.

Solution. Extend the existing guard seam with role-aware predicates — canArchiveCustomer(role), canEditCustomer(customer, role) — sourced from the same Clerk session use-customer-owner already reads. Pure functions, trivially testable, gate the UI while RLS stays the real backstop.

Clerk claims (role) ──▶ guards.ts ──▶ row-actions renders only permitted verbs
                        └▶ RLS still rejects the write (backstop unchanged)

Benefits. Locality: every "who can do what" rule lives beside the data it guards, unit-tested without a browser. Leverage: the pattern drops into every module's guards.ts.

Quick wins Speculative

  • Drop axios. It's in package.json but CLAUDE.md forbids raw fetch for Supabase — likely dead weight. Delete it unless something outside this module uses it.
  • Cap the toast array. toast-store.ts appends unboundedly; trim to the last N so a burst of errors can't grow the list forever.
  • Reconsider blanket force-dynamic. 2026 App Router expects Server Components / Server Actions for reads; the whole app being client-fetched is a legitimate style, but it's a deliberate trade worth recording as an ADR rather than a default.
Top recommendation

Start with Candidate 1 — server-side list querying.

It's the one thing standing between "nice demo" and "production." It's a true depth move (behaviour goes behind the service interface, the view gets simpler), it activates the CustomerFilters type that already exists, and it sets the read shape every later module copies. Candidate 2 (DB-owned time) is the natural follow-on in the same service file. Candidates 3–5 are quality-of-life and can wait until module #2 proves the duplication is real.
