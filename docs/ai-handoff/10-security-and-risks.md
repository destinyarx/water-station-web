# Security & Risks

**This is a static code inspection, not a penetration test.** No requests were
sent to a running server, no auth was actually attempted/bypassed, and no
dependency scanner was run. Findings are based on reading source, ADRs, and
`docs/DATABASE.md`. **The absence of a finding here does not mean absence of
risk** — it means this pass did not surface one. Treat every row's
"Confidence" column literally, and re-verify anything marked **Requires
validation** before relying on it.

[`09-frontend-and-ux.md`](./09-frontend-and-ux.md). For the full repo map,
see [`05-codebase-map.md`](./05-codebase-map.md). For per-table RLS policy
text, see `docs/DATABASE.md` (not reproduced in full here).

## Notes on scope and method

- Evidence was gathered by reading source files directly (services, hooks,
  guards, middleware, ADRs, `docs/DATABASE.md`), not by running the
  application, attempting authentication bypasses, or executing any
  dependency/SAST scanner.
- Every RLS policy claim above is sourced from `docs/DATABASE.md`'s written
  policy tables, cross-referenced against ADRs — **not** from querying the
  live Supabase project's actual `pg_policies`. Given there is no
  `supabase/migrations/` folder in this repo (schema is applied manually per
  the dashboard), a documentation/reality drift is structurally possible and
  is called out explicitly above rather than assumed away.
- No secret values were read, printed, or inferred at any point — `.env`'s
  existence and gitignore status were confirmed without opening the file.
