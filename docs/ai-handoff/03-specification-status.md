# Specification Status — Traceability Report

Cross-checks every spec folder (`docs/specs/000`–`013`) and every ADR against the
actual code in `src/`. Status values: **Implemented**, **Partially
implemented**, **Planned**, **Deprecated**, **Contradicted by implementation**,
**Unable to verify**. Feature specs are traced in the main table below; every
ADR gets its own explicit status in the dedicated
[ADR-by-ADR traceability](#adr-by-adr-traceability) section further down. See
[`05-codebase-map.md`](05-codebase-map.md) for file-tree detail and
[`07-data-architecture.md`](07-data-architecture.md) for full schema; this file
does not repeat either. See also [`04-feature-map.md`](04-feature-map.md) for a
feature-by-feature inventory of current implementation (roles, entry points,
file structure) that complements this traceability report.

Every non-trivial claim below is labeled **Confirmed** (I read the cited file),
**Inferred** (reasoned from adjacent evidence), **Unknown**, **Potentially
outdated**, or **Requires validation**.

## Unresolved decisions

1. **Was ADR 0003's revert capability deliberately retired by feature 010, or is it a regression?** No ADR or spec note answers this. (See deliveries row above.)
2. **What is the actual RLS policy for `expenses`?** `docs/DATABASE.md`'s expenses section is a checklist template, never filled in with real policy text.
3. **Does the customers UPDATE RLS policy still require `created_by = jwt.sub` with no owner override**, and if so, is the missing owner-check in `customers.guards.ts`/`customer-row-actions.tsx` an intentional gap or an oversight (products has the equivalent owner-override; customers does not)?
4. **Is `/dashboard` and `/sales` intentionally left as stubs pending a future spec**, or is a spec simply missing? CONTEXT.md's "Planned modules" list includes "Reports" but not "Dashboard" or "Sales" by name (sales appears only as a nav item and a business-domain product-type distinction in CONTEXT.md, not a module in its own right).
5. **When will AquaFlow AI move off the mock endpoint?** No spec or ADR gives a timeline or a follow-up issue for wiring the real Gemini-backed edge function.
6. **Resolved 2026-07-15:** legal consent is ADR 0014 and the organizations architecture filename is padded to ADR 0003; ADR 0009 now unambiguously means the organization UUID decision.
7. **`public.delivery_schedule_dates` (feature 010) uses a different RLS identity pattern than every other table** — its policy is written against a `public.users`/`org_id` join keyed on `auth.uid()`, while every other organization-owned table in this schema authorizes via `auth.jwt() ->> 'sub'` (the Clerk-claims pattern from ADR 003/0009). No note explains why this table diverges. See `07-data-architecture.md §5.1` for the full detail. Confirm whether this was intentional (e.g. copied from a different reference project) or an oversight before building more on top of this table.
