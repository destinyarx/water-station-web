## Quick wins

Low-effort, low-risk, no product decision required:

- **Q16** — add a `typecheck` script.
- **Q19** — cap the toast queue length.
- **Q14** — renumber the duplicate ADR `0009`.
- **Q02** — reconcile the `org_id` type contradiction in `docs/DATABASE.md` (once the live column type is confirmed — the fix itself is a doc edit).
- **Q01** — the *verification* step (run dev, check console) is a quick win even though the potential fix is not to be done unilaterally.

## Structural improvements (bigger, but foundational)

- **Q04** — server-side filtering/pagination in list services. This is the one change every future module inherits, per the architecture review's own top recommendation.
- **Q09** — a shared mutation-hook factory, once a third near-identical hook is about to be written (the review calls this the point where the abstraction "earns its keep").
- **Q05** — a CI workflow. Nothing else on this list is enforced automatically today; this is the highest-leverage single addition for keeping the rest from regressing silently.

## Risks of change

- **Q01** (proxy/middleware) — if the file turns out to be inactive, renaming it to `middleware.ts` is simple, but it will suddenly start enforcing auth/onboarding redirects that may have silently not applied for a while. Test the full sign-in → onboarding → protected-route flow (all roles) immediately after, not just the happy path.
- **Q10** (table pattern) — retrofitting TanStack Table onto shipped, working list UIs risks visual/behavioral regressions in modules users already rely on daily (customers, products). Prefer doing this feature-by-feature alongside otherwise-planned UI work, not as a standalone mass rewrite.
- **Q12** (transactional delivery services) — moving to a Postgres RPC changes the error surface (Postgres function errors vs. current service-level error mapping); update `documents`/`deliveries` error-handling and manual RLS/verification docs together.
- **Q11** (Zustand vs hand-rolled stores) — whichever direction is chosen, do it once and update all three root docs (`CLAUDE.md`, `docs/ARCHITECTURE.md`, `docs/CODING_STANDARDS.md`) in the same change so they don't re-drift.

## Improvements needing a product decision

- **Q03** (document storage) — needs a decision on file size limits, retention, and whether mobile camera capture (see `13-mobile-handoff.md`) is in scope before designing the bucket/path structure once instead of twice.
- **Q06** (error monitoring vendor) — needs budget/vendor/data-residency decision (PH-based small-business customers).
- **Q10** (table pattern) — needs a design decision: adopt the documented TanStack Table standard everywhere, or formally amend the standard to match the shipped inline-styled grid look.
- **Q17** (component testing investment) — needs a decision on whether manual QA (already required by `docs/TESTING.md` for UI states) is sufficient or whether interaction bugs (e.g. the two-step confirm-dialog flow) justify adding a rendering test harness.

## Improvements needing runtime profiling before acting

- **Q20** (query cache tuning) — requires real request-volume/latency data from a deployed environment; no profiling tooling exists in the repo today (no APM/analytics, per Q06).
- **Q12** (transactional writes) — the *urgency* of fixing this depends on how often two staff members concurrently edit the same delivery/maintenance occurrence in practice; worth instrumenting before investing in a full RPC rewrite.

## Areas already strong — preserve, do not "fix"

- **Tenant isolation model**: RLS as the actual security boundary, with `org_id`/`created_by` always resolved from Clerk claims via dedicated `use-*-owner` hooks, never from form input. Confirmed consistent across customers/products/deliveries/maintenance/documents/notifications/AquaFlow AI. This is the single most important property of the codebase — any refactor must preserve it.
- **Four-shape type model + mapper** (`Row` / `Display` / `FormInput`/`FormValues` / `Insert`/`Update`), kept in one mapper file per feature so DB↔UI conversion can't drift. The architecture review's own "deletion test" (would removing this make call sites worse?) says keep it.
- **Generic two-step confirm-dialog pattern** (`useSubmitConfirm` + `SaveConfirmDialog` + `useMutationDialog`) reused by every feature's create/edit/archive dialogs — genuinely deep, not boilerplate.
- **Service-layer error handling discipline**: every service function throws a friendly constant string on `{ error }`, never surfaces raw Postgres text — verified consistently across every feature service read this pass.
- **`axios` is not dead weight** (correcting a claim in `docs/improve-architecture/architecture-review.html`'s quick wins): it is used exclusively for the two onboarding Supabase Edge Function calls (`src/features/registration/services/registration.service.ts`, `use-complete-registration.ts`), not for normal Supabase database queries — this is compliant with `CLAUDE.md`'s "no raw fetch for normal Supabase database queries" rule, since Edge Functions are not normal DB queries. Do not remove it.
- **Test suite is 100% green and fast**: 48 files / 208 tests in ~9s (`npm run test`, this pass). `npx tsc --noEmit` and `npm run lint` are both clean (0 errors; 13 pre-existing warnings, see [`12-testing-and-operations.md`](./12-testing-and-operations.md)). Whatever else changes, keep this bar.
